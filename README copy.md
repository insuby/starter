### 1.2. Файл обмена данными

#### 1.2.1. Требования к файлу

Файл ДОЛЖЕН иметь кодировку UTF-8 и расширение `*.json`.

В самом верху файлы ДОЛЖНЫ быть поля:
* created_at - дата создания файла и его отправки на брокер. ДОЛЖЕН быть в формате ISO;
* actual_at - дата актуальности информации. ДОЛЖЕН быть в формате ISO;
* version - версия файла обмена данными. ДОЛЖЕН быть строкой вида "1.0";
* author - кодовое имя управления: "upr-1", "upr-2", "upr-3", "upr-4", "upr-5". ДОЛЖЕН быть строкой;
* data - поле в котором будут расположены все данные. ДОЛЖЕН быть массивом.

Поле "version" необходимо для поддержания версионности файла. При небольших изменениях изменяется вторая цифра. При больших изменениях, несовместимых с прошлыми версиями, повышается первая цифра. Начинаем с "1.0".

Поле "author" необходимо для определения источника данных. Также, по нему будет определятся набор атрибутов.

#### 1.2.2. Атрибуты

Атрибут представляет собой структуру для описания элемента. Для каждого такого элемент

Атрибут ДОЛЖЕН иметь уникальное имя (колонка atr) для управления. На каждое управление будет отдельная таблица атрибутов и полей.

Список всех атрибутов управления ДОЛЖЕН быть представлен в таблице атрибутов (SQL запрос на вставку/обновление атрибутов).

##### 1.2.2.1. Как составить атрибут?

* Это данные объекта? -> сразу добавляем object_code
* Эта структура данных повторяется на других слайдах? -> атрибут становиться типом array
* Это простая одиночная цифра? -> тип num
* Это одиночная цифра, но с какой-то дополнительной информацией? -> тип object

#### 1.2.3. Поля

Поле представляет собой "колонку атрибута". Если у нас есть статистика по "отправлено", "норма" и другие, то именно это и будут поля атрибута "Статистика за объект".

Поля МОГУТ быть в оригинальном именовании как в базах управлений.

Поля ДОЛЖНЫ иметь перевод в таблице полей.

Поля object_code и parent_object_code ДОЛЖНЫ быть у атрибутов с типом array для установления иерархии подчинения.

Список всех полей атрибутов управления ДОЛЖЕН быть представлен в таблице полей (SQL запрос на вставку/обновление полей)

#### 1.2.2. Наполение данными

> Поля "name", "type" самих атрибутов можно удалить из файла, так как по таблице атрибутов (пример в excel) можно будет понять что это за атрибут и как с ним работать.

Таблица атрибутов (в excel) - пример атрибутивного состава. Для каждого атрибута задается тип и обязательные поля (ключи). Таким образом, можно выстроить структуру атрибутов разработчикам непосредственно управлений (а то и сохранить свою структуру и нам предоставить наименования ключей и их значение).

К таблице атрибутов привязана таблица полей.
Основные поля в ней:
* atr - ссылка на атрибут;
* key - поле данного атрибута;
* key_ru - перевод данного поля.

### 1.3. Отправка и деление файла

В файле при необходимости МОЖНО передавать не все атрибуты за управление. То есть часть атрибутов МОЖНО передать сейчас и часть потом. Делить атрибуты на файлы МОЖНО по-своему усмотрению.
